**Claude**

\*\*Desktop Duplication API (DXGI Output Duplication)\*\*입니다.

Windows 8부터 도입됐고, Windows 10에서는 mirror driver 방식이 사실상 퇴출되면서 원격제어/화면캡처의 표준이 됐습니다.

- `IDXGIOutputDuplication` 인터페이스로 GPU가 합성한 최종 프레임버퍼를 직접 획득
- **dirty rects / move rects** 메타데이터를 제공해서 변경 영역만 전송 가능 (mirror driver의 hook 방식 대체)
- 프레임이 GPU 텍스처(`ID3D11Texture2D`)로 나오므로 GPU 상에서 바로 인코딩(NVENC 등)이나 압축 파이프라인 연결 가능

참고로 Windows 10 1903+에서는 **Windows.Graphics.Capture (WGC)** API도 추가돼서 창 단위 캡처가 필요하면 이쪽을 씁니다. 지금 작업 중인 dirty-rect 타일 diffing 파이프라인이라면 DDA가 주는 dirty rect 메타데이터를 1차 필터로 쓰고 그 위에 타일 단위 해싱을 얹는 구조가 일반적입니다.

### 왜 그 순서여야 하는가

DXGI가 프레임을 재구성하는 규약 자체가 **move 먼저 → dirty 나중** (move rect 배열은 주어진 순서대로)입니다. 즉:

```
cur_frame ≡ apply_dirty(apply_moves(prev_frame, move_rects), dirty_rects)
```

따라서 타일 비교의 기준(reference)은 원본 prev가 아니라 **move가 적용된 prev** 여야 합니다:

```python
def process_frame(prev, cur, move_rects, dirty_rects):
    # Stage 0: move를 prev에 순서대로 적용 + 클라이언트에 명령 전송
    for m in move_rects:                      # 배열 순서 엄수 (겹치는 move 존재 가능)
        blit(prev, src=m.source_point, dst=m.destination_rect)
        send_copyrect(m)                      # 좌표만 전송

    # Stage 1~2: 이제 prev는 "move 반영된 상태" → 이걸 기준으로 타일 비교
    for tile in tiles_covering(dirty_rects):
        if not tile_equal(prev, cur, tile):
            mark_changed(tile)

    # Stage 4 후: prev를 cur로 갱신 (통째로 swap하거나 변경 타일만 복사)
```

핵심은 **서버의 prev와 클라이언트의 프레임버퍼가 항상 동일 상태** 를 유지해야 한다는 것. 서버가 prev에 move를 적용하고 클라이언트도 같은 CopyRect를 실행하면 양쪽이 동기화되고, 그 위에 dirty 타일 delta를 얹으면 됩니다. (XOR delta 인코딩을 쓴다면 delta의 기준도 이 move-적용된 prev입니다.)

### 실무 주의점 3가지

1. **겹치는 move**: source와 destination이 겹치는 move(예: 1px 스크롤)는 blit 방향에 따라 결과가 달라지므로 `memmove` 시맨틱(방향 인지 복사)으로 처리해야 합니다.
2. **move 후 dirty 중복**: DDA는 move한 영역의 일부를 dirty로도 같이 보고하는 경우가 흔합니다(스크롤 후 새로 노출된 하단 영역 등). 이건 버그가 아니라 정상이고, 위 파이프라인에서 자연스럽게 처리됩니다 — move 적용 후 타일 비교에서 실제로 다른 부분만 살아남으니까요. 오히려 move가 정확했다면 move 영역 위의 dirty 타일 상당수가 검증 단계에서 탈락합니다.
3. **move가 손해인 경우**: destination 영역이 어차피 dirty로 대부분 덮이는 경우 CopyRect는 클라이언트 왕복 연산만 낭비합니다. 휴리스틱으로 `move 면적 대비 dirty와 겹치는 비율 > ~80%` 면 move를 버리고 그냥 dirty로 흡수시키는 게 낫습니다. 반대로 대형 스크롤(문서/브라우저)에서는 move 하나가 수 MB를 좌표 8바이트로 바꿔주는 최대 절감 포인트입니다.

### Windows

#### 1\. CreateDIBSection — GDI 호환 + 직접 픽셀 접근 (가장 일반적)

BitBlt 등 GDI 연산이 가능하면서 픽셀 포인터를 직접 얻을 수 있는 유일한 GDI 비트맵:

```c
BITMAPINFO bmi = {0};
bmi.bmiHeader.biSize        = sizeof(BITMAPINFOHEADER);
bmi.bmiHeader.biWidth       = width;
bmi.bmiHeader.biHeight      = -height;      // 음수 = top-down (원격제어 필수)
bmi.bmiHeader.biPlanes      = 1;
bmi.bmiHeader.biBitCount    = 32;           // BGRA
bmi.bmiHeader.biCompression = BI_RGB;

void* pixels = NULL;
HBITMAP hbmp = CreateDIBSection(NULL, &bmi, DIB_RGB_COLORS, &pixels, NULL, 0);

// pixels로 직접 memcpy/비교 가능, 동시에 HDC에 SelectObject해서 BitBlt도 가능
HDC memDC = CreateCompatibleDC(NULL);
SelectObject(memDC, hbmp);
```

`biHeight` 를 음수로 주는 것이 포인트입니다. 양수면 bottom-up이라 DDA 프레임과 행 순서가 뒤집힙니다.

#### 2\. D3D11 Staging Texture — DDA 파이프라인용

DDA로 얻은 GPU 텍스처를 CPU로 읽으려면 이쪽이 정석:

```c
D3D11_TEXTURE2D_DESC desc = {0};
desc.Width  = width;   desc.Height = height;
desc.MipLevels = 1;    desc.ArraySize = 1;
desc.Format = DXGI_FORMAT_B8G8R8A8_UNORM;
desc.SampleDesc.Count = 1;
desc.Usage          = D3D11_USAGE_STAGING;
desc.CPUAccessFlags = D3D11_CPU_ACCESS_READ;

ID3D11Texture2D* staging;
device->CreateTexture2D(&desc, NULL, &staging);

// DDA 텍스처 → staging → Map
context->CopyResource(staging, ddaTexture);   // 또는 CopySubresourceRegion으로 dirty rect만
D3D11_MAPPED_SUBRESOURCE mapped;
context->Map(staging, 0, D3D11_MAP_READ, 0, &mapped);
// mapped.pData, mapped.RowPitch — RowPitch != width*4 일 수 있음 주의
```

전체 `CopyResource` 대신 `CopySubresourceRegion` 으로 dirty rect 영역만 내리면 PCIe 대역폭도 절약됩니다.

#### 3\. Raw 버퍼 — prev frame 저장용

GDI 연산이 필요 없는 순수 비교/delta 버퍼라면 이게 가장 빠릅니다:

```c
// 64바이트 정렬 → AVX-512 memcmp 친화적
uint8_t* prev = _aligned_malloc((size_t)stride * height, 64);
```

move rect blit도 `memmove` 행 단위로 직접 구현하면 되므로 HBITMAP이 필요 없습니다.

### macOS

#### 1\. CGBitmapContextCreate — Quartz 메모리 비트맵 (DIBSection 대응)

```c
size_t stride = width * 4;
void* pixels = malloc(stride * height);

CGColorSpaceRef cs = CGColorSpaceCreateDeviceRGB();
CGContextRef ctx = CGBitmapContextCreate(
    pixels, width, height,
    8,                  // bits per component
    stride,
    cs,
    kCGImageAlphaPremultipliedFirst | kCGBitmapByteOrder32Little  // = BGRA
);
// pixels 직접 접근 + CGContextDrawImage 등 Quartz 그리기 둘 다 가능
```

`kCGBitmapByteOrder32Little + AlphaPremultipliedFirst` 조합이 메모리상 BGRA로, Windows DIB와 동일한 레이아웃이 되어 크로스 플랫폼 코드 공유에 유리합니다.

#### 2\. CVPixelBuffer — ScreenCaptureKit/VideoToolbox 파이프라인용

SCK 프레임이 이 형태로 오고, VideoToolbox 인코더 입력도 이 형태입니다:

```swift
var pb: CVPixelBuffer?
CVPixelBufferCreate(
    kCFAllocatorDefault, width, height,
    kCVPixelFormatType_32BGRA,
    [kCVPixelBufferIOSurfacePropertiesKey: [:]] as CFDictionary,  // IOSurface-backed
    &pb
)

CVPixelBufferLockBaseAddress(pb!, .readOnly)
let base   = CVPixelBufferGetBaseAddress(pb!)
let stride = CVPixelBufferGetBytesPerRow(pb!)   // width*4가 아닐 수 있음
// ... 타일 비교 ...
CVPixelBufferUnlockBaseAddress(pb!, .readOnly)
```

IOSurface-backed로 만들면 GPU와 zero-copy 공유가 되어 하드웨어 인코딩 연결 시 복사가 없습니다.

#### 3\. Raw 버퍼

Windows와 동일하게 prev frame은 `posix_memalign(&p, 64, stride*height)` 정렬 버퍼가 가장 단순하고 빠릅니다.

### 파이프라인 기준 권장 조합

| 용도 | Windows | macOS |
| --- | --- | --- |
| 캡처 프레임 수신 | D3D11 staging texture (Map) | SCK의 CVPixelBuffer (Lock) |
| prev frame / 타일 비교 | 64B 정렬 raw 버퍼 | 64B 정렬 raw 버퍼 |
| GDI/Quartz 그리기 필요 시 | CreateDIBSection | CGBitmapContextCreate |
| GPU 인코더 직결 | D3D11 texture 그대로 (NVENC) | IOSurface-backed CVPixelBuffer (VT) |

공통 주의점 하나: 양쪽 다 **stride(RowPitch/BytesPerRow)가 width×4와 다를 수 있으므로** 타일 비교·memcpy 루프는 반드시 행 단위로 stride를 타야 합니다.
